也许您负责开发基于容器的应用程序基础架构？如果是这样，您很可能了解容器可以为开发人员、架构师和运营团队提供的价值。事实上，您可能一直在阅读容器，并对更详细地探索该技术感到兴奋。但是，在深入讨论容器在生产环境中的体系结构和部署之前，开发人员、架构师和系统管理员需要了解三件重要事项：

所有应用程序（包括容器化应用程序）都依赖于底层内核
内核通过系统调用为这些应用程序提供 API
此 API 的版本控制很重要，因为它是确保用户空间和内核空间之间确定性通信的"粘合剂"
虽然容器有时被视为虚拟机，但重要的是要注意，与虚拟机不同，内核是程序和它们需要访问的资源之间的唯一抽象层。让我们看看为什么。

好了，您了解了什么是进程，容器就是进程，但是容器映像中的文件和程序呢？这些文件和程序构成了所谓的用户空间。启动容器时，程序将从容器映像加载到内存中。容器中的程序运行后，仍需要对内核空间进行系统调用。用户空间和内核空间以确定性方式进行通信的能力至关重要。

用户空间
用户空间是指位于内核外部的操作系统中的所有代码。大多数类Unix操作系统（包括Linux）都预先打包了各种实用程序，编程语言和图形工具 - 这些都是用户空间应用程序。我们经常将其称为"用户空间"。

Userland应用程序可以包括用C，Java，Python，Ruby和其他语言编写的程序。在容器化世界中，这些程序通常以容器映像格式（如 Docker）交付。当您从红帽注册表下拉并运行红帽企业 Linux 7 容器映像时，您正在利用预打包的最小红帽企业 Linux 7 用户空间，其中包含 bash、awk、grep 和 yum 等实用程序（以便您可以安装其他软件）。
```
docker run -i -t rhel7 bash
```
所有用户程序（无论是否容器化）都通过操作数据来运行，但是这些数据位于何处？此数据可以来自 CPU 和外部设备中的寄存器，但最常见的是存储在内存和磁盘上。用户程序通过向内核发出称为系统调用的特殊请求来访问数据。示例包括分配内存（变量）或打开文件。内存和文件通常存储不同用户拥有的敏感信息，因此必须通过系统调用从内核请求访问。

内核空间
内核为安全性、硬件和内部数据结构提供了抽象。open（） 系统调用通常用于获取 Python、C、Ruby 和其他语言的文件句柄。您不希望程序能够对 XFS 文件系统进行位级更改，因此内核提供系统调用并处理驱动程序。实际上，此系统调用非常常见，是 POSIX 库的一部分。

请注意，在手册页中，某些系统调用（也称为接口）已在不同版本的内核中添加和删除。Linus Torvalds et.要非常小心地保持这些系统调用的行为被很好地理解和稳定。从Red Hat Enterprise Linux 7（内核3.10）开始，有382个系统调用可用。不时添加新的系统调用，弃用旧系统调用;在考虑容器基础架构的生命周期以及将在其中运行的应用程序时，应考虑这一点。

结论
您需要了解一些关于用户空间和内核空间的重要要点：

应用程序包含业务逻辑，但依赖于系统调用。
编译应用程序后，应用程序使用的一组系统调用（即依赖）嵌入在二进制文件中（在更高级的语言中，这是解释器或JVM）。
容器不会抽象出用户空间和内核空间共享一组公共系统调用的需求。
在容器化的世界中，此用户空间被捆绑在一起并运送到不同的主机，从笔记本电脑到生产服务器。
在未来几年，这将带来挑战。
随着时间的推移，保证今天构建的容器将在明天的容器主机上运行将是一项挑战。想象一下，今年是2024年（也许我们最终会拥有真正的悬浮滑板），你仍然有一个基于容器的应用程序，需要在生产环境中运行红帽企业Linux 7用户空间。如何安全地升级底层容器主机和基础架构？容器化应用程序在市场上任何最新的容器主机上都能同样出色地运行吗？
